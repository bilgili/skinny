## Context

Both backends run the same `main_pass.slang` megakernel; the renderer is
backend-agnostic and branches on `ctx.is_metal` only where the two genuinely
differ. The Vulkan path uses `-fvk-use-scalar-layout` (tight `float3` = 12 B) and
binds resources through descriptor sets; the native Metal path compiles in
SlangPy/RHI with MSL layout (`float3` padded to 16 B) and binds every resource
**by name at dispatch** (`_build_metal_binds`), with no descriptor sets
(`descriptor_sets is None`).

Two existing mechanisms already handle the MSL `float3` hazard: fixed structs
(`FlatMaterialParams`, `BvhNode`) are **float4-wrapped** in the Slang source, and
the gen-reflected skin UBO is **relocated** field-by-field from scalar offsets to
reflected MSL offsets (`_pack_mtlx_skin_array_msl`, design D3). The
MaterialX-generated `GraphParams_*` structs are neither — they are emitted by
MaterialXGenSlang with raw `float3` fields and were packed/uploaded in scalar
layout, and their per-graph SSBOs were never added to `_build_metal_binds`.
Result: any graph that reads its params (not a bound bindless texture) sees a
zeroed/misaligned buffer; the purely-procedural marble (`fractal3d` → colour
params) goes black.

Separately, several host paths were written Vulkan-first and reach
`ctx.command_pool` / `queue_family_indices` / `vk.VkDescriptorBufferInfo` without
a Metal branch — latent crashes that only fire on specific scenes (20+ instances
triggering a buffer realloc) or GUI actions (opening the debug viewport, picking
a BXDF surface).

## Goals / Non-Goals

**Goals:**
- Procedural and graph-driven std_surface materials render on Metal at the same
  perceptual parity bar the skin path already meets.
- No host path crashes on the compute-only Metal context; Vulkan-only features
  either no-op (descriptor rebinds) or degrade visibly (GUI tools).
- Reuse the established design-D3 relocation pattern rather than inventing a new
  one; keep the Vulkan path byte-identical.

**Non-Goals:**
- Native Metal ports of the GUI visualisers (debug viewport, BXDF/BSSRDF) — they
  degrade for now; a Metal tool-dispatch sibling is later work.
- Graph-internal `SamplerTexture2D` parity: a textured graph (wood) is ~10 %
  off on Metal because its texture is sampled through a sampler field inside the
  graph struct, a different binding path — tracked as a separate follow-up.
- Wavefront-backend parity (ReSTIR/neural/indirect dispatch) — covered by the
  separate `metal-wavefront-parity` change.

## Decisions

- **Relocate graph params, don't float4-wrap them.** The `GraphParams_*` structs
  are generated by MaterialXGenSlang and would have to be post-processed to
  float4-wrap. Instead reflect each buffer's MSL element layout
  (`ComputePipeline.graph_param_layouts`: `{field: (offset, size)}` + stride) and
  copy each field from its scalar offset to its MSL offset at the MSL stride —
  the exact mechanism `_pack_mtlx_skin_array_msl` uses. Falls back to the scalar
  pack when reflection is unavailable. *Alternative considered:* patching the
  generated Slang to emit float4 — rejected as fragile against regen.
- **Bind graph params by name in `_build_metal_binds`.** Mirror every other
  resource: `b[f"graphParams_{gf.sanitized_name}"] = buf.buffer`. The pipeline
  filters to referenced graphs, so over-binding is harmless. This is the missing
  half — relocation alone is useless if the buffer is never bound.
- **Guard at the method, not the call site.** `_update_texture_pool_descriptors`
  and the three rebind helpers get an `is_metal` early return, covering all
  current and future callers. Metal's `_build_metal_binds` re-reads the live
  buffer every dispatch, so a realloc needs no rebind.
- **Degrade GUI tools through their existing fallback.** The BXDF visualiser
  already has a zeroed-grid path for wavefront mode (`pipeline is None`); route
  Metal through it (`pipeline is None or is_metal`). The debug viewport raises a
  clear `RuntimeError` its Qt dock already catches.
- **Parity coverage must use a graph scene.** Add a gated test driving a
  multi-material USD scene; assert whole-frame rel-MSE + a per-material
  "not near-black on one backend only" guard. *Why gated:* it compiles the
  megakernel on Metal (MTLCompilerService RAM spike) and builds a Vulkan device.

## Risks / Trade-offs

- [Reflected MSL offsets drift if SlangPy changes its layout rules] → offsets
  come from live reflection every build, never a hand table; the relocation is a
  no-op fallback to scalar when reflection is missing.
- [Over-binding unused graph buffers] → harmless: the compiled module references
  a subset; slang-rhi ignores unbound-but-provided names.
- [Wood (graph-internal texture) still ~10 % off] → out of scope here; whole-
  frame parity stays within bar because the textured material is dim. Tracked
  separately.
- [GUI features silently unavailable on Metal] → acceptable; they fail loud
  (debug viewport message) or visibly (empty BXDF grid), never crash.

## Migration Plan

Already shipped to `main` as six commits (gpu_info+sRGB, texture-pool no-op,
graph-param bind+MSL, rebind no-ops, debug-viewport refusal, BXDF degrade). This
change formalises the spec delta retroactively; archiving folds the requirement
into `openspec/specs/metal-backend`. No data migration; rollback is a revert of
those commits (Metal-only, Vulkan untouched).
