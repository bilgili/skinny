## Requirements
### Requirement: Session-fixed spectral mode selection

The renderer SHALL provide an opt-in spectral render mode selected at startup
via a `--spectral` CLI flag on all front-ends (env var `SKINNY_SPECTRAL`), fixed
for the session like the execution mode, and not persisted. Spectral mode SHALL
support the `path`, `bdpt`, `sppm`, and `mlt` integrators in their existing
execution envelopes on flat-material scenes. The spectral `path` integrator SHALL admit
the analytic directional-proposal sets `{bsdf}`, `{bsdf,env}`, and `{env}` in
both megakernel and wavefront execution modes. Startup SHALL refuse the flag
with a clear error when combined with ReSTIR DI reuse, the neural directional
proposal, or a scene requiring subsurface/skin or heterogeneous-volume
transport. When the flag is absent, behavior SHALL be identical to the RGB
pipeline.

#### Scenario: spectral analytic environment proposal starts

- **WHEN** the app starts with `--spectral --integrator path --proposals
  bsdf,env` (or `env`) in either execution mode
- **THEN** the session runs the spectral path integrator with the requested
  analytic proposal set and reports that set as resolved, without a spectral
  pin

#### Scenario: unsupported spectral layers remain refused

- **WHEN** `--spectral` is combined with a neural proposal or ReSTIR DI reuse
- **THEN** startup fails before GPU initialization with an error naming the
  unsupported combination

#### Scenario: interactive neural preset resolves safely

- **WHEN** an interactive spectral session selects a preset containing the
  neural proposal
- **THEN** the renderer removes the neural proposal, retains any supported
  analytic proposal in the preset, falls back to BSDF if none remains, and
  reports the pin in resolved configuration

#### Scenario: spectral environment proposal is path-only

- **WHEN** an explicit spectral BDPT/SPPM/MLT launch requests an environment
  proposal, or an interactive spectral session switches from path with an
  environment proposal selected to BDPT/SPPM/MLT
- **THEN** explicit startup is refused with an error naming `path` as the
  required integrator, while an interactive switch resolves to native BSDF
  sampling and reports the pin

### Requirement: RGB build stays byte-identical

The spectral mode SHALL be implemented as a compile-time shader variant (`SKINNY_SPECTRAL`)
such that the default (non-spectral) Vulkan SPIR-V is byte-identical to the pre-change
binary, and the default Metal build compiles from unchanged effective source.

#### Scenario: default SPIR-V unchanged

- **WHEN** the megakernel is compiled without the spectral define after the change lands
- **THEN** the produced `main_pass.spv` is byte-identical to the checked-in pre-change binary
  (verified by a hostless test or recorded compile check)

### Requirement: Hero-wavelength spectral transport in the path integrator

In spectral mode the path integrator SHALL carry radiance and throughput as 4 spectral
samples: one hero wavelength drawn per camera path from the visible-wavelength importance
distribution over [360, 830] nm, plus 3 equal-spaced rotated secondaries, each with its
sampling pdf. Transport SHALL run in the megakernel execution mode on both backends (Vulkan
and Metal), covering flat materials, analytic lights, emissive triangles, and the
environment. The wavelengths drawn at path start SHALL be used consistently at every bounce
and at the film resolve of that path.

#### Scenario: backends agree spectrally

- **WHEN** the same spectral render runs on Vulkan and on native Metal
- **THEN** the images agree within the recorded backend-parity tolerance

#### Scenario: kernel math matches the CPU mirror

- **WHEN** the wavelength-sampling, upsampling, and film-resolve GPU kernels run against the
  numpy reference implementation on shared inputs
- **THEN** outputs match within the kernel-test tolerance

### Requirement: RGB assets render via spectral upsampling

In spectral mode, RGB-authored inputs SHALL be upsampled to spectra on the GPU using the
Jakob–Hanika sigmoid-polynomial model: reflectance-mode coefficients for material parameters
and textures, and an illuminant-mode variant (coefficients scaled by the tabulated CIE D65
SPD) for RGB-specified lights and environment-map radiance. The coefficient table SHALL be
generated by a checked-in port of pbrt's `rgb2spec` optimizer, validated against
pbrt-published coefficient values in a hostless test, and uploaded as a storage buffer (no
new sampled texture).

#### Scenario: upsampling round-trip preserves color

- **WHEN** an sRGB reflectance value is upsampled to a spectrum and integrated back through
  the CIE pipeline under an equal-energy illuminant
- **THEN** the recovered linear sRGB matches the input within the round-trip tolerance
  recorded by the hostless table test

#### Scenario: existing RGB scene renders under spectral mode

- **WHEN** a corpus scene authored entirely with RGB values and textures is rendered with
  `--spectral`
- **THEN** the render completes using upsampled spectra and passes its recorded parity gates

#### Scenario: no new texture binding on Metal

- **WHEN** the spectral session initializes on native Metal
- **THEN** the upsampling table binds as a storage buffer and the compute argument-table
  texture count does not increase

### Requirement: Spectral film resolve feeds the existing accumulation pipeline

At path termination the spectral estimate SHALL resolve per sample to XYZ using the same
analytic CIE colour-matching-function fit the importer uses (Wyman–Sloan–Shirley), divided by
the per-wavelength sampling pdf and normalized by the CIE Y integral, then convert to linear
sRGB with the existing matrix and enter the existing clamp → RGBA32F progressive accumulation.
Exposure, tonemap, sRGB display, readback, and linear-HDR sampling for the parity harness
SHALL be unchanged.

#### Scenario: downstream pipeline untouched

- **WHEN** a spectral render is read back via `render_headless()` and via the linear
  accumulation path
- **THEN** both use the same buffer formats and code paths as an RGB render (only pixel
  values differ)

#### Scenario: film clamp still applies

- **WHEN** a spectral sample resolves to a color exceeding `fc.filmMaxComponent`
- **THEN** the existing per-sample radiance clamp bounds it identically to the RGB pipeline

### Requirement: Exact spectral sources

Where authored source data is spectral, spectral mode SHALL evaluate it exactly rather than
via RGB upsampling: blackbody emitters evaluate Planck's law at the sampled wavelengths from
the preserved temperature; named conductor Fresnel evaluates densely resampled spectral eta/k
curves; authored illuminant SPDs (light `L`/`I` spectra) evaluate densely resampled SPDs from
a shared spectral-asset storage buffer. Authored *reflectance* spectra MAY continue to reduce
through RGB in this change (recorded limitation).

#### Scenario: blackbody emitter is spectral

- **WHEN** a scene lights geometry with an imported `blackbody` emitter under `--spectral`
- **THEN** emission is computed from Planck's law at the sampled wavelengths (not the
  imported RGB), and the rendered chromaticity moves toward the pbrt reference relative to
  the RGB render

#### Scenario: conductor Fresnel is spectral

- **WHEN** a named metal (e.g. `metal-Au`) renders under `--spectral`
- **THEN** Fresnel uses spectral eta/k at the sampled wavelengths rather than the 3-primary
  RGB approximation

#### Scenario: authored illuminant SPD survives to the GPU

- **WHEN** a light authors a non-constant `spectrum L` and the scene renders under
  `--spectral`
- **THEN** the light's emission at each sampled wavelength comes from the resampled authored
  SPD, not from an upsampled RGB reduction

### Requirement: Dispersion for wavelength-dependent dielectrics

Dielectrics SHALL refract using the IOR evaluated at the hero wavelength when their imported
IOR is wavelength-dependent (named glasses with preserved Cauchy/Sellmeier fits), and the
first dispersive refraction SHALL terminate the secondary wavelengths with the pbrt
secondary-termination pdf adjustment, so the estimator remains unbiased.

#### Scenario: prism disperses

- **WHEN** a dispersive dielectric is lit by a narrow white source under `--spectral`
- **THEN** the transmitted light spatially separates by wavelength (visible spectrum), which
  the RGB pipeline cannot produce

#### Scenario: non-dispersive dielectrics keep 4-wavelength transport

- **WHEN** a constant-IOR dielectric is rendered under `--spectral`
- **THEN** no secondary termination occurs and all 4 wavelength samples contribute to the
  film resolve

### Requirement: Furnace closure holds under spectral mode

The white-furnace closure gate SHALL pass in spectral mode: a lossless material inside the
constant furnace environment (upsampled to a spectrum) SHALL vanish into the background
within the existing uniformity tolerance, and resolved sample colors SHALL be non-negative
after the gamut clamp.

#### Scenario: spectral furnace closes

- **WHEN** a furnace-suite lossless-material scene renders under `--spectral`
- **THEN** the furnace uniformity metric passes the same gate as the RGB mode

### Requirement: Spectral mode respects Metal dispatch hygiene

Spectral megakernel dispatches SHALL stay within the established watchdog bounds (row-band
megakernel tiling under `SKINNY_METAL`), and the change SHALL pass the Metal kill harness
since kernel length changes.

#### Scenario: kill harness passes with spectral kernels

- **WHEN** `tests/test_metal_cleanup.py` (hostless tests and the gpu-marked harness under the
  guarded runner) runs on the change
- **THEN** all cleanup/kill probes pass with spectral mode exercised in the probe render

### Requirement: Spectral path transport supports the environment-importance proposal

The spectral path integrator SHALL sample non-delta bounce directions through
the shared directional-proposal seam. For `{bsdf,env}` and `{env}`, proposal
selection and pdf evaluation SHALL remain scalar and wavelength-independent,
while the chosen direction's BSDF response and environment radiance SHALL be
evaluated at the path's hero wavelengths. Spectral throughput SHALL divide by
the full proposal-mixture solid-angle pdf, and NEE, environment-miss,
emissive-hit, and sphere-hit MIS SHALL use that same generating density. The
environment proposal SHALL reuse the existing environment CDF resources and add
no GPU state.

#### Scenario: spectral environment mixture stays unbiased

- **WHEN** an IBL-lit flat-material scene is rendered with `--spectral
  --integrator path --proposals bsdf,env`
- **THEN** each non-delta bounce divides its per-wavelength response by
  `α_bsdf·p_bsdf + α_env·p_env`, and the converged result matches spectral
  BSDF-only transport within the recorded stochastic tolerance

#### Scenario: both execution modes use the same proposal contract

- **WHEN** the same spectral path scene and analytic proposal set render under
  megakernel and wavefront execution
- **THEN** both execution modes sample through `sampleBounceDirection`, recolor
  the response per hero wavelength, and use the returned mixture pdf for
  downstream MIS

#### Scenario: BSDF-only spectral behavior is preserved

- **WHEN** spectral path tracing runs with the default `{bsdf}` proposal
- **THEN** the shared proposal sampler takes its BSDF-only fast path and the
  spectral response/pdf estimator is unchanged
